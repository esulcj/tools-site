Gastown Deep Dive: How to Actually Use It

1. How Does the Mayor Know What to Do?

The Mayor is a Claude Code session with full access to gt commands. It's not a classifier or a router. It's a persistent AI agent running in ~/gt/mayor/ that:

  • Reads incoming mail/nudges as natural language
  • Has access to all 42+ formulas via bd formula list
  • Has the full town context via gt prime (which injects rig state, bead state, polecat availability, memories)
  • Makes judgment calls about which formula to use

The mechanism is prompt-based. When the Mayor gets gt prime, it sees:

  • Its role identity (Mayor)
  • All registered rigs and their status
  • Pending beads/issues
  • Available polecats
  • Injected memories (via gt remember)
  • The formula checklist for its current patrol

How experienced users communicate with the Mayor:

  • gt nudge mayor "description of what you want" — for quick requests
  • gt mail send mayor --subject "BUILD: X" --body "detailed goal" — for formal requests
  • The Mayor reads the request, picks the formula, creates beads, and slings to polecats

There is no classifier. The Mayor is an LLM making decisions. The quality of its decisions depends on:

  1. How clear your request is
  2. What memories you've stored (gt remember)
  3. What the formula checklist tells it to do
  4. What it learned from previous sessions (via seance, memories, bead history)

2. How Does Gastown Self-Improve?

Memory System (gt remember / gt memories)

Persistent key-value store backed by beads. Memories are injected during gt prime for every agent.

# Store a lesson
gt remember "Refinery uses worktree, cannot checkout main"
gt remember --type feedback "Don't mock the database in integration tests"
gt remember --type user --key senior-go-dev "User has 10 years Go experience"

# Search/list memories
gt memories                    # All memories
gt memories --type feedback    # Behavioral corrections only
gt memories refinery           # Search term

Memory types:

  • feedback — Corrections from users (behavioral guidance)
  • project — Ongoing work context, goals, deadlines
  • user — Info about the user
  • reference — Pointers to external resources
  • general — Uncategorized

Key insight: This replaces CLAUDE.md-based memory with bead-backed storage. Every gt prime injects relevant memories. So if you gt remember --type feedback "Always run tests before gt done", every polecat will see that on their next prime.

Bead History

The beads system tracks:

  • What work was done (bead descriptions, notes)
  • Verdicts from the Refinery (pass/fail/retry)
  • Which polecats did what
  • Time taken, issues found

Agents can query this history. The Mayor sees all beads when it primes.

Seance (Learning from Dead Sessions)

gt seance                           # List predecessor sessions
gt seance --talk <id> -p "What did you find?"  # Query a dead session

New agents can literally ask their predecessors what they learned. This is how context survives session death.

Hooks (Lifecycle Events)

Hooks fire at key moments:

  • pre-compact-prime — before context compaction, preserve critical info
  • post-verdict — after Refinery verdict, could trigger learning
  • post-done — after polecat completion

What Gastown Does NOT Do Automatically

  • Does not write CLAUDE.md files. That's our addition.
  • Does not create project-specific lessons. Memories are town-wide, not rig-specific.
  • Does not automatically learn from failures. You have to gt remember the lesson manually, or set up hooks to do it.

How to Close the Learning Loop

  1. After each build: gt remember --type feedback "<lesson>" — injected into all future primes
  2. For project-specific lessons: Write to rig CLAUDE.md (our pattern, not native Gastown)
  3. For behavioral corrections: gt remember --type feedback is the strongest signal
  4. For debugging past decisions: gt seance --talk <id> to query dead sessions

3. Production Setup

Proper Installation Sequence

# 1. Install prerequisites
brew install gastown dolt beads tmux

# 2. Create town
gt install ~/gt --git
cd ~/gt

# 3. Verify installation
gt doctor --fix

# 4. Start everything properly
gt up                    # Starts daemon + all rig services
# OR for more control:
gt daemon start          # Just the daemon
gt rig boot <name>       # Start specific rig services

gt up vs gt daemon start

  • gt up — Starts daemon AND boots ALL registered rigs (witness, refinery, polecats)
  • gt daemon start — Starts ONLY the daemon process
  • gt rig boot <name> — Boots a specific rig's services

For our setup: gt up is the right command. It brings everything online.

Rig Configuration

Each rig can be configured in its directory:

  • CLAUDE.md — Agent instructions (standard Claude Code convention)
  • .beads/config.yaml — Beads configuration
  • Dolt database for persistent state

Mayor Configuration

The Mayor's behavior is configured via:

  1. Memories — gt remember stores behavioral guidance
  2. Town config — mayor/town.json (name, version)
  3. Formula availability — what's in .beads/formulas/
  4. Its own CLAUDE.md — at ~/gt/mayor/CLAUDE.md (if present)

Common Pitfalls

  1. Not running gt doctor --fix — catches missing settings.json, stale formulas, broken config
  2. Outdated formulas — gt doctor reports "10 outdated, 5 new". Need to update.
  3. Large repos as rigs — cloning is slow. Use small focused repos.
  4. Not using gt up — starting daemon without booting rigs leaves witness/refinery dead
  5. Bypassing the Mayor — manually creating beads and slinging defeats the purpose
  6. Not storing memories — the system can't learn if you don't gt remember

The Complete Lifecycle

Human has idea
    ↓
Tycho (main) writes goal document
    ↓
gt mail send mayor --subject "BUILD: X" --body "<goal>"
    ↓
Mayor reads goal, picks formula (mol-idea-to-plan / shiny / etc)
    ↓
Mayor creates beads, sets dependencies
    ↓
Mayor slings beads to available polecats
    ↓
Polecats work (each runs gt prime → sees formula checklist → works → gt done)
    ↓
Refinery processes MRs (verify → merge to main OR reject → re-dispatch)
    ↓
Mayor reports completion (mail to overseer/human)
    ↓
Tycho deploys from merged main branch
    ↓
gt remember --type feedback "<lessons from this build>"

Key Formulas

Formula Purpose When to Use
mol-idea-to-plan Vague idea → approved PRD → beads New features with unclear requirements
shiny Design-first engineering (Engineer in a Box) Standard builds with clear goals
shiny-enterprise Shiny + Rule of Five (5 parallel reviews) High-stakes builds
shiny-secure Shiny + security audit Security-sensitive work
mol-polecat-work Standard work lifecycle Simple task execution
mol-polecat-code-review Code review pipeline Reviewing existing code
mol-polecat-review-pr PR review GitHub PR reviews
mol-digest-generate Status digest generation Daily/weekly summaries
mol-convoy-feed Auto-dispatch ready work Batch processing
rule-of-five 5 parallel reviews of same work Quality-critical work

Dogs (Infrastructure Agents)

Dogs are cross-rig workers dispatched by the Deacon:

  • alpha — Triage dog, handles escalations
  • boot — Startup/health checks
  • Custom dogs can be added for maintenance tasks

For our use case: Dogs could run deploy health checks, email monitoring, cron-equivalent tasks.
