1. The Failure Stats

Before the architecture diagrams, the sobering numbers. Research from arXiv 2503.13657 (the MAST taxonomy) studied multi-agent LLM system failures systematically:

41-87% Failure rate in production multi-agent systems
79% Failures from specification + coordination (not infra)
~16% Failures from infrastructure (what everyone obsesses over)
7x Accuracy gain from structured validation (PwC/CrewAI)

The failure taxonomy breaks down as:

CategoryShareExamples
Specification41.8%Role ambiguity, unclear task boundaries, missing constraints
Coordination36.9%Lost context across handoffs, conflicting decisions, state sync failures
Verification21.3%No validation, hallucination cascades, premature termination
Infrastructure~16%Rate limits, context overflow, cascading timeouts
The counterintuitive insight: Most multi-agent system builders spend 80% of effort on infrastructure. But 80% of failures come from vague role definitions and broken agent-to-agent communication. The fix is boring: treat agent specs like API contracts, not prose.

2. The Five Orchestration Patterns

Every multi-agent system in the wild maps to one of five patterns, or a hybrid. These show up across all frameworks.

Pattern 1: Thin Router (Triage + Handoff) Most Proven

A lightweight classifier receives all input, routes to the right specialist, and gets out of the way. The specialist owns the conversation from there.

User → Router → [Specialist A | Specialist B | Specialist C]
                 ↓
              Direct response to user

Who uses it: OpenAI Swarm (the entire framework is this pattern), OpenClaw's agent routing, Google ADK's "Coordinator" pattern, most customer service bots.

Why it works: Minimal context overhead. Each specialist gets a clean, focused prompt. Router is cheap (can use a small model). Specialists don't interfere with each other.

Why it breaks: Tasks that span multiple domains force re-routing or multi-specialist coordination. Router misclassification sends tasks to the wrong specialist with no recovery path. Context from the routing decision is often lost.

Implementation (OpenAI Agents SDK):

triage = Agent(
    name="Triage",
    instructions="Route customer queries to the right specialist.",
    handoffs=[billing_agent, tech_support_agent, general_agent]
)
# Specialist becomes the active agent after handoff
# No manager narrating — specialist responds directly

Pattern 2: Orchestrator-Workers (Manager + Parallel Specialists) High Potential, Complex

A central orchestrator LLM dynamically decomposes tasks, spawns worker agents in parallel, and synthesizes results. The orchestrator retains control throughout.

User → Orchestrator ──┬── Worker A (research)
                       ├── Worker B (code)
                       └── Worker C (review)
                       ↓
              Orchestrator synthesizes → Response

Who uses it: Claude Code (internally), OpenClaw's main session pattern, LangGraph's subgraph delegation, Anthropic's recommended "orchestrator-workers" workflow.

Why it works: Flexible decomposition. Parallelism. Orchestrator can adapt strategy based on intermediate results.

Why it breaks: Orchestrator context bloats with every worker's output. Synthesizing bad worker results produces confident garbage. Orchestrator becomes a bottleneck. Cost scales with coordination overhead.

Pattern 3: Pipeline (Sequential Chain) Reliable for Defined Tasks

Each agent handles one step and passes structured output to the next. No backtracking, no dynamic routing.

Input → Agent A → Agent B → Agent C → Output
        (draft)    (review)   (polish)

Who uses it: Anthropic's "prompt chaining" pattern, most content generation pipelines, CrewAI's sequential process.

Why it works: Predictable, debuggable, each agent has minimal context. Easy to add programmatic gates between steps. Cost and latency are linear, not exponential.

Why it breaks: Can't handle tasks requiring iteration or backtracking. Error in step 1 cascades through all subsequent steps with no recovery. Over-engineering simple tasks with unnecessary pipeline stages.

Pattern 4: Hierarchical Teams Seductive, Dangerous

Multiple layers of management. A top-level coordinator delegates to team leads, who delegate to specialist workers. Like a corporate org chart, but for LLMs.

CEO Agent
  ├── Team Lead: Engineering
  │     ├── Frontend Worker
  │     └── Backend Worker
  └── Team Lead: Research
        ├── Web Searcher
        └── Analyst

Who uses it: CrewAI's hierarchical process, AutoGen's nested chat, enterprise setups trying to mirror their org structure.

Why it works: Intuitive mapping to human organizations. Good separation of concerns at team boundaries.

Why it breaks: Every layer of delegation loses context, adds latency, and multiplies cost. Middle managers (team leads) add overhead without value. The "hallucination cascade" is devastating: a small error at a leaf node becomes gospel for the parent, who passes it up as fact. Debugging across 3+ layers is nearly impossible.

The hallucination cascade: A minor hallucination from a "Research Agent" becomes accepted fact for the "Execution Agent," causing small errors to snowball into complete system failure. This is the single biggest killer of hierarchical multi-agent systems. (Source)

Pattern 5: Swarm / Peer Mesh Research Stage

Autonomous agents with no central coordinator. Each agent decides independently when to act and who to hand off to. Emergent behavior through local rules.

Agent A ←→ Agent B
  ↕           ↕
Agent C ←→ Agent D
(Each agent can hand off to any other)

Who uses it: Academic research, some robotics applications. Not production LLM systems.

Why it works (in theory): No single point of failure, infinitely scalable, emergent intelligence.

Why it breaks (in practice): LLMs are not reliable enough for unsupervised peer coordination. Without a coordinator, agents duplicate work, enter infinite handoff loops, or collectively decide to do nothing. No one owns the final answer.

3. Framework Comparison

The major frameworks each optimize for different patterns. None is universally best.

Framework Core Pattern Best For Weakness
LangGraph Graph-based state machine Complex workflows with branching, checkpoints, human-in-the-loop State management overhead. Steep learning curve. Overkill for simple tasks.
CrewAI Role-based teams (Crews + Flows) Quick assembly of role-based workflows. Fastest setup. Struggles with adaptability. Free-form text output without output_pydantic.
AutoGen Conversational agents Human-in-the-loop, collaborative research, brainstorming Too many LLM calls for simple tasks. Overkill for extraction/pipeline work.
OpenAI Swarm Handoffs + Routines Customer service routing, triage patterns Educational only (no official support). No persistent state. Stateless between calls.
OpenAI Agents SDK Agents-as-tools + Handoffs Production systems needing both delegation patterns Lightweight. Less deep orchestration than LangGraph.
Google ADK Coordinator + Context Compaction Production multi-agent with built-in context management Google ecosystem lock-in. Newer, less battle-tested.
OpenClaw Thin router + specialist spawning Personal assistant multi-agent, file-based persistence, channel routing Context bloat in long sessions. Compaction is lossy. No structured inter-agent protocol.
Key insight from Anthropic: "The most successful implementations weren't using complex frameworks or specialized libraries. Instead, they were building with simple, composable patterns." Start with augmented LLM calls. Add agents only when single-agent approaches provably fail. (Source)

4. The Context Problem

Every multi-agent system eventually hits the same wall: context management. This is the hardest unsolved problem in the space.

Context Rot

As tokens increase in the context window, recall accuracy decreases. This isn't a bug, it's an architectural property of transformers. Every token attends to every other token (n² relationships). As n grows, attention gets stretched thin. Models develop their attention patterns from training data where shorter sequences are more common.

Anthropic's context engineering guide frames this as an "attention budget" problem: every new token depletes the budget, increasing the need for curation.

Compaction: The Lossy Compression Trap

When context gets too large, systems compact it: summarize the history and replace it with the summary. Every framework does this differently:

SystemCompaction ApproachRisk
OpenClawLLM-generated summary replaces old messagesSummary grows with each compaction. Critical details silently dropped. "Compaction spiral" where summaries of summaries lose signal.
Google ADKSession-layer context compactionSame core risk, but built into the framework with explicit controls.
Claude CodeJust-in-time loading via lightweight referencesRequires tool calls to re-load context. Slower but more accurate.
Microsoft Agent FrameworkToken limit triggers automatic compactionHard cutoff can lose mid-conversation context.
The compaction spiral: Agent runs for a while → context fills up → system compacts (summarizes) → summary loses important details → agent makes mistakes because it forgot things → generates more tokens trying to recover → context fills up again → compacts again → summary of a summary loses even more → repeat until the agent is essentially amnesiac.

Anthropic's Recommended Solutions

  1. Compaction with guidelines: Don't just "summarize." Use explicit compression guidelines that specify what to keep (current task, recent decisions, next steps) and what to drop.
  2. Structured note-taking: Agent writes structured notes to external files as it works. When context is compacted, the notes survive. This is what Claude Code does — scratchpads and TODO files are context insurance.
  3. Multi-agent decomposition: Instead of one agent with a huge context, spawn specialists with clean, minimal context for each subtask. Each agent starts fresh. The orchestrator holds the minimal state needed to coordinate.
  4. Just-in-time loading: Don't keep everything in context. Store references (file paths, query IDs, URLs) and load data only when needed via tool calls.

5. What Anthropic Actually Recommends

Anthropic published two canonical guides that every multi-agent builder should read:

Building Effective Agents (Dec 2024)

The core argument: start simple, add complexity only when needed.

  1. Start with a single augmented LLM — tools, retrieval, memory. Most tasks don't need multiple agents.
  2. Add workflow patterns in order of complexity:
    • Prompt chaining (sequential steps)
    • Routing (classify → specialist)
    • Parallelization (same task, multiple perspectives)
    • Orchestrator-workers (dynamic decomposition)
    • Evaluator-optimizer (generate + critique loop)
  3. Only use autonomous agents when the task genuinely requires dynamic planning and can't be solved with a predefined workflow.

Effective Context Engineering (2025)

The core argument: context is a finite resource with diminishing returns.

  • System prompts: Find the "Goldilocks zone" — specific enough to guide behavior, flexible enough to handle edge cases. Not brittle if-else logic, not vague hand-waving.
  • Tools: Minimal viable set. If a human engineer can't tell which tool to use in a given situation, an LLM definitely can't.
  • Examples: Curate diverse canonical examples, not a laundry list of edge cases. Examples are "pictures worth a thousand words."
  • Dynamic retrieval: Move from pre-loaded context to just-in-time tool-based loading. Maintain lightweight references, load on demand.

6. OpenClaw Multi-Agent Configs

OpenClaw supports multi-agent routing natively. Here's how people set it up:

The OpenClaw Pattern

// openclaw.json — agent routing
{
  "agents": {
    "main": {
      "model": "anthropic/claude-sonnet-4-5",
      "systemPrompt": "You are Coordinator. Route tasks...",
      "workspace": "~/.openclaw/workspace"
    },
    "coder": {
      "model": "anthropic/claude-sonnet-4-5",
      "systemPrompt": "You are a coding specialist...",
      "workspace": "~/projects"
    },
    "researcher": {
      "model": "anthropic/claude-sonnet-4-5",
      "systemPrompt": "You are a research specialist...",
      "workspace": "~/.openclaw/workspace/research"
    }
  },
  "routing": {
    "discord": {
      "default": "main",
      "channels": {
        "coding-help": "coder",
        "research": "researcher"
      }
    }
  }
}

openclaw-agents Community Kit

The openclaw-agents repo (348 stars) provides a one-command setup with 9 specialized agents: Engineer, Writer, Researcher, Designer, DevOps, Data Analyst, Security, QA, and a Coordinator. Each agent gets its own identity, workspace files, and routing rules.

Key design choices:

  • Channel-based routing: Different Discord channels route to different agents automatically
  • Per-agent workspaces: Each agent has isolated file access
  • Spawning for delegation: Main agent uses sessions_spawn for build tasks, keeping the conversation session clean
  • File-based persistence: Agent lessons, project state, and memory stored in markdown files that survive session boundaries

7. What We're Doing and Why It Breaks

Honest assessment of our specific failure modes, mapped to the industry patterns.

🔴 Compaction Spirals

Pattern: Main session runs for hours across Discord. Context fills up. Compaction fires. Summary tries to preserve everything. Next compaction summarizes the summary. After 3-4 compactions, the agent has forgotten its active tasks.

Industry name: Context degradation / context collapse.

What others do: Claude Code uses scratchpad files. Google ADK has Session-layer compaction with explicit retention rules. LangGraph uses graph checkpoints that can be replayed.

Our mitigation: compaction-discipline rule: max 4000 chars, state lives in files not summaries, no accumulating historical decisions. Better than nothing, but it's an advisory rule enforced by the same context that's being compacted. It's the fox guarding the henhouse.

🔴 Orchestrator Context Bloat

Pattern: Main session is both the conversational interface AND the orchestrator. It accumulates tool call results, sub-agent outputs, file reads, and user messages in the same context window. A research task fills half the context before the user asks their next question.

Industry insight: This is the #1 reason Anthropic recommends multi-agent decomposition — "spawn specialists with clean, minimal context for each subtask." The orchestrator should hold minimal state.

Our mitigation: orchestrator-discipline rule: never write >500 chars of code inline, delegate to sub-agents. Works for builds, but research results and tool outputs still bloat the main session.

🟡 Specification Drift

Pattern: Sub-agents receive task specs as natural language strings in sessions_spawn. Over time, the specs get more detailed and contradictory as we add rules to fix edge cases. A sub-agent might receive 2000 words of instructions, half of which conflict.

Industry name: The 41.8% specification problem from MAST.

What others do: JSON schema for agent specifications. CrewAI's structured task definitions with output_pydantic. Formal role contracts.

Our approach: Markdown files (CLAUDE.md, AGENTS.md) that agents read on startup. Good for persistence, bad for schema validation. Nothing prevents contradictions.

🟡 No Inter-Agent Protocol

Pattern: Sub-agents communicate results back via natural language in their final message. No structured schema. The orchestrator parses prose to understand what happened. If a sub-agent says "I deployed the site" — did it? Was there an error? What URL?

Industry fix: Structured communication protocols (MCP's JSON-RPC, typed message envelopes with explicit status codes).

Our gap: Sub-agent results are auto-announced as plain text. No schema, no status codes, no structured handoff. Orchestrator has to interpret prose.

🟢 What's Actually Working

  • Thin router for channel routing: Discord channels → specific agents works reliably
  • File-based persistence: CLAUDE.md lesson accumulation, project STATUS files, and scratchpads survive across sessions. This is our equivalent of Claude Code's scratchpad pattern.
  • Max depth 2 delegation: Limiting to Main → Worker (no Main → Planner → Worker) avoids the hierarchical team failure mode
  • Sub-agent isolation: Each spawned agent gets a clean context. No inherited bloat from the parent session.
  • Label-based tool restriction: Deploy agents need label="deploy" for Cloudflare access. Security agents get different capabilities. This is primitive but effective tool restriction.

8. Hard-Won Lessons

Synthesized from research, community reports, and our own experience. These apply across frameworks.

Architecture

  1. Start with one agent. Add agents only when you can prove a single agent fails at the task. Most people add agents too early. (Anthropic, repeatedly)
  2. Two levels of delegation max. Every additional layer multiplies context loss, cost, and debug difficulty. Hierarchical teams are seductive and deadly.
  3. Thin router + fat specialists beats fat router + thin specialists. The router should be dumb and fast. Specialists should have focused prompts and relevant tools. Not the reverse.
  4. Parallel beats sequential for independent subtasks. If workers don't depend on each other, run them simultaneously. Sequential chaining adds latency and creates cascading failure points.

Context Management

  1. Context is a finite resource with diminishing returns. Not "more context = better." Every token competes for attention. Curate ruthlessly.
  2. State belongs in files, not in context. Anything important enough to survive should be written to disk. Context is working memory. Disk is long-term memory.
  3. Just-in-time loading beats pre-loading. Store references, load on demand. Don't stuff the entire codebase into context "just in case."
  4. Compaction is lossy compression. Design around this reality. Don't rely on compaction to preserve critical information.

Agent Specifications

  1. Treat agent specs like API contracts, not documentation. Required fields, explicit constraints, success criteria. Vague specs → vague behavior.
  2. One agent, one responsibility. An agent that does research AND writes code AND reviews PRs will do all three poorly.
  3. Tool sets should be minimal and unambiguous. If you can't tell which tool to use, neither can the LLM. Overlapping tools cause confused behavior.

Failure Handling

  1. Add an independent validator. A judge agent that evaluates outputs against specs catches hallucination cascades before they ship. PwC saw 7x accuracy improvement from this alone.
  2. Every handoff must include full context. Stateless handoffs (like Swarm) force you to be explicit about what the next agent needs. This is a feature, not a limitation.
  3. Monitor for infinite loops. Agents can enter retry spirals or circular handoffs. Hard iteration limits are essential, not optional.
  4. Errors compound, not cancel. A 5% error rate per agent means a 23% error rate across 5 sequential agents (1 - 0.95⁵). Design for error isolation.
