Part 1: Architecture

Gastown is a workspace manager that coordinates multiple AI coding agents (Claude Code, Codex, Copilot, Gemini, Cursor, others) working in parallel on different tasks. It persists work state in git-backed hooks, enabling reliable multi-agent workflows. Think of it as Kubernetes for coding agents.

Core Principles

PrincipleMeaning
MEOW (Molecular Expression of Work)Break goals into detailed, trackable, atomic units that agents execute autonomously
GUPP (Gas Town Universal Propulsion Principle)"If there is work on your Hook, YOU MUST RUN IT." Agents proceed without waiting.
NDI (Nondeterministic Idempotence)Useful outcomes through orchestration of potentially unreliable processes. Persistent beads + oversight agents guarantee eventual completion.

Component Map

ComponentLevelRoleInterface
TownRootYour workspace directory (~/gt/). Contains everything.Directory
Mayor 🎩TownAI coordinator. Primary interface. Claude Code instance with full workspace context. Picks formulas, creates beads, slings to polecats.gt mayor attach
DeaconTownCross-rig supervisor daemon. Continuous patrol cycles. Monitors health, dispatches Dogs, escalates unresolvable issues.gt patrol
Dogs 🐕TownInfrastructure workers dispatched by the Deacon for maintenance tasks. Boot Dog checks Deacon every 5 min.gt dog
Rig 🏗️ProjectProject container. Wraps a git repo + manages its agents.gt rig add
Polecat 🦨RigWorker agent. Persistent identity, ephemeral sessions. Has permanent agent bead, CV chain, work history. Works in isolated git worktrees.Spawned by Mayor
WitnessRigPer-rig lifecycle manager. Monitors polecats, detects stuck agents, triggers recovery, manages session cleanup.Automatic patrol
Refinery 🏭RigPer-rig merge queue processor. Bors-style bisecting queue. Polecats never push directly to main.Automatic
Crew 👤RigLong-lived named agents for persistent collaboration. Unlike polecats, maintain context across sessions.gt crew add
Convoy 🚚WorkWork tracking unit. Bundles related beads. Mountain convoys get autonomous stall detection.gt convoy
Bead 📿WorkAtomic work item. Hash-based IDs (prefix + 5-char: gt-abc12). Dependency tracking, auto-ready detection.bd commands
Molecule 🧬WorkWorkflow template instance. Formulas (TOML) instantiated as molecules with tracked steps.bd mol commands

Three-Tier Watchdog System

Daemon (Go process) ← heartbeat every 3 min └── Boot (AI agent) ← checks Deacon every 5 min └── Deacon (AI agent) ← continuous patrol └── Witnesses & Refineries ← per-rig agents

Multiple agents observing overlapping state is intentional redundancy. If the Witness misses something, the Deacon or Daemon catches it.

Communication Patterns

MethodCommandUse Case
Mailgt mail send <target>Async messages between agents. Creates bead entries. Primary coordination primitive.
Nudgegt nudge <target>Real-time messaging. Immediate communication without mail system.
Escalategt escalate -s <severity>Severity-routed issue escalation. Agent → Deacon → Mayor → Overseer.

The Bead Lifecycle

bd create "Task" ↓ open bd update --claim ↓ in_progress (hooked to polecat) polecat works (gt prime → formula checklist → execute) ↓ gt done (pushes branch, creates MR bead) ↓ done → refinery Refinery batches MRs → verification gates ↓ pass → squash-merge to main ↓ fail → bisect, isolate failing MR, re-dispatch Merged / Rejected

Session-Per-Step Model

Polecats do NOT complete complex molecules end-to-end. Each molecule step gets one polecat session. The sandbox (branch, worktree) persists across sessions. Sessions are the pistons; sandboxes are the cylinders.

ConceptTriggerWhat ChangesWhat Persists
Session cycleHandoff, compaction, crashClaude context windowBranch, worktree, molecule state
Step cycleStep bead closedCurrent step focusBranch, worktree, remaining steps

First-Time Setup

# 1. Create the Town
gt install ~/gt --git
cd ~/gt

# 2. Enable system-wide and initialize
gt enable
gt git-init

# 3. Add a project (rig)
gt rig add myproject https://github.com/you/repo.git

# 4. Create crew workspace (optional, for human work)
gt crew add yourname --rig myproject

# 5. Start all services
gt up    # Starts daemon + boots ALL rigs (witness, refinery, polecats)

# 6. Health check
gt doctor --fix
gt status

What gt doctor --fix Checks

  • settings.json sync: Verifies all .claude/settings.json files match what gt hooks sync would generate
  • Formula freshness: Reports outdated or missing formulas (e.g., "10 outdated, 5 new")
  • Dolt health: Validates database connectivity and integrity
  • Beads status: Checks .beads/ directories and JSONL integrity
  • Hooks configuration: Ensures hooks are properly wired for all agents
  • tmux availability: Verifies tmux for full-stack mode

Part 3: The Mayor Pipeline

The Mayor is a Claude Code session with full access to gt commands. It's not a classifier or router. It's a persistent AI agent running in ~/gt/mayor/ that reads incoming requests, has access to all 42+ formulas, and makes judgment calls.

Communicating with the Mayor

# Formal request (preferred for builds)
gt mail send mayor --subject "BUILD: Dashboard" --body "Goal: Build a real-time monitoring dashboard..."

# Quick nudge
gt nudge mayor "Fix the CSS on the login page"

# Attach directly (interactive session)
gt mayor attach

# Check Mayor's inbox
gt mail inbox

How to Structure Build Requests

I need to...Subject prefixMayor picks...
Build something newBUILD: <name>shiny (design-first) or mol-polecat-work
Research/plan a vague ideaRESEARCH: <name>mol-idea-to-plan
Review code/PRsREVIEW: <name>mol-polecat-code-review or mol-polecat-review-pr
Generate status digestDIGESTmol-digest-generate
DO NOT manually run bd create or gt sling. That's the Mayor's job. You write the goal, mail it, and relay human questions. The Mayor picks formulas, creates beads, and slings to polecats. Bypassing the Mayor is bypassing the enforcement mechanism.

The mol-idea-to-plan Pipeline

Mail arrives: "RESEARCH: <name>" ↓ Mayor selects mol-idea-to-plan ↓ Phase 1: PRD (6 polecats explore requirements in parallel) ↓ synthesize Phase 2: Design (6 polecats produce design docs in parallel) ↓ synthesize Phase 3: Review rounds (iterate until approved) ↓ Phase 4: Bead decomposition (create work items from approved plan) ↓ Phase 5: Sling to polecats for implementation

The shiny Pipeline

Design-first engineering pipeline. Used for standard builds with clear goals. The Mayor receives a well-structured goal, creates beads with dependencies, and slings them to polecats. Each polecat sees the formula checklist inline via gt prime and works through it step-by-step.

Escalation Flow

Polecat hits blocker ↓ gt escalate -s <SEVERITY> "description" ↓ Deacon receives ├── resolves → updates issue, re-slings work └── cannot → forwards to Mayor ├── resolves → updates issue, re-slings └── cannot → forwards to Overseer (human)
SeverityPriorityRoute
CRITICALP0bead + mail + email + SMS
HIGHP1bead + mail + email
MEDIUMP2bead + mail to Mayor

Stale escalations (unacknowledged past 4h) auto-bump severity (MEDIUM→HIGH→CRITICAL) up to 2 times.

Relaying Human Questions

# Check for Mayor escalations
gt mail inbox

# Relay answer back
gt mail send mayor --subject "RE: question about auth flow" --body "Use OAuth2 with PKCE..."

# Acknowledge an escalation (prevents re-escalation)
gt escalate ack <bead-id> --note "Investigating"
The communication bridge:
Theo → Main (Tycho) → Mayor → Polecats ← Mayor (escalations) ← Main → Theo
Main (Tycho) is the Theo-facing interface. Mayor is the build orchestrator. Tycho does NOT do the Mayor's job.

Part 4: Memory System

Persistent key-value store backed by beads. Memories are injected during gt prime for every agent. This is how the system learns.

Memory Types

TypeFlagPurposeExample
Feedback--type feedbackBehavioral corrections from users. Strongest signal."Always run tests before gt done"
Project--type projectOngoing work context, goals, deadlines"Dashboard v2 ships by April 1"
User--type userInfo about the user"User prefers light theme, no frameworks"
Reference--type referencePointers to external resources"API docs at docs.example.com/v3"
General(default)Uncategorized"Refinery uses worktree, cannot checkout main"

Commands

# Store a lesson
gt remember "Refinery uses worktree, cannot checkout main"
gt remember --type feedback "Don't mock the database in integration tests"
gt remember --type user --key senior-go-dev "User has 10 years Go experience"

# Search/list memories
gt memories                    # All memories
gt memories --type feedback    # Behavioral corrections only
gt memories refinery           # Search by term

How Memories Get Injected

Every time an agent runs gt prime, it sees:

  • Its role identity (Mayor, Polecat, Witness, etc.)
  • All registered rigs and their status
  • Pending beads/issues
  • Available polecats
  • Injected memories (from gt remember)
  • The formula checklist for its current patrol/work

Best Practices: Memory vs CLAUDE.md

Goes in gt rememberGoes in Rig CLAUDE.md
Town-wide behavioral correctionsProject-specific rules and constraints
User preferences across all projectsData schemas, API contracts
Lessons from Refinery verdictsBuild commands, test commands
General process guidanceDesign tokens, file structure
"Always X when Y" patterns"This project uses X framework with Y config"
Gastown does NOT automatically learn from failures. You must gt remember lessons manually, or set up hooks to do it. Memories are town-wide, not rig-specific. For rig-specific lessons, write to the rig's CLAUDE.md.

Seance: Learning from Dead Sessions

# List predecessor sessions
gt seance

# Query a dead session
gt seance --talk <id> -p "What did you find?"

# Full context conversation with predecessor
gt seance --talk <id>

New agents can literally ask their predecessors what they learned. Context survives session death through seance.

Part 5: The Refinery

The Refinery is a per-rig merge queue processor. It implements a Bors-style bisecting merge queue. Polecats never push directly to main.

How MRs Get Processed

Polecat calls gt done ├── Pushes branch to origin ├── Creates MR bead (label: gt:merge-request) ├── Sends POLECAT_DONE mail to witness └── Session exits ↓ Witness receives POLECAT_DONE ├── Checks cleanup_status (trust polecat self-report) ├── If clean → sends MERGE_READY to refinery └── If dirty → creates cleanup wisp ↓ Refinery processes MERGE_READY ├── Claims MR (sets assignee) ├── Acquires merge slot (serialized push lock) ├── Runs quality gates ├── Squash-merges to main ├── Closes MR bead and source issue └── Sends MERGED mail to witness

Verdict Types

VerdictWhat HappensRecovery
PassAll MRs in batch merge to mainN/A - success
FailBisects to isolate the failing MRGood MRs merge; bad MR isolated and re-dispatched or escalated
RetryTransient failure detectedMR re-queued for next batch

Failure Recovery

FailureDetectionRecovery
gt done fails mid-executionWitness DetectZombiePolecats()Finds stuck-in-done, recovers
POLECAT_DONE mail lostWitness patrol finds dead session with hook_beadZombie detection and recovery
Merge conflictRefinery doMerge()Creates conflict resolution task, blocks MR
MERGED mail lostWitness finds closed bead with live sessionZombie detection
Known Issue: In our first builds, the Refinery didn't auto-merge. We had to manually assemble from worktrees. Investigate Refinery configuration if this persists.

The Self-Improvement Loop

Build completes ↓ Refinery verdict (pass/fail) ↓ If fail: analyze what went wrong ↓ gt remember --type feedback "<lesson from failure>" ↓ Next build: gt prime injects that memory ↓ Agent avoids the same mistake

Hook Points for Automated Learning

Hook PointWhen It FiresSelf-Improvement Use
pre-compact-primeBefore context compactionPreserve critical info. Agent can bd update --notes before losing context.
post-verdictAfter Refinery verdictTrigger gt remember based on verdict. Auto-capture failure patterns.
post-doneAfter polecat completionRecord what worked. Append to CLAUDE.md if project-specific.
The feedback loop in practice:
1. After each build: gt remember --type feedback "<lesson>"
2. For project-specific lessons: write to rig CLAUDE.md
3. For debugging past decisions: gt seance --talk <id>
4. For behavioral corrections: gt remember --type feedback is the strongest signal.

A spec is NOT a schema. A spec tells the agent exact paths, exact commands, exact error cases. If an agent has to guess, your spec failed. Goals → User Stories → Specifications → Contracts → Beads. Don't skip steps.

Bead Decomposition Patterns

PatternBeadsWhen
Minimal2 (data + frontend)Simple apps with clear data flow
Standard3-4 (data + frontend + state/tests)Apps with state management or API layer
Full6-8 (data + API + frontend + auth + tests + deploy)Complex multi-component systems

Dependencies matter: 2 parallel + 1 serial (data beads first, frontend depends on data) is better than 3 parallel if frontend needs data schema output.

Parallel Wave Patterns

Wave 1 (parallel): [Data Layer] [Config/Schema] ↓ both complete Wave 2 (parallel): [Frontend] [API Endpoints] [Tests] ↓ all complete Wave 3 (serial): [Integration / Final Assembly]

The Gastown-Gate Plugin

OpenClaw enforces the Mayor pipeline via the gastown-default override in AGENTS.md. ALL multi-file builds go through Gastown. No exceptions. The process:

  1. Read agents/domain/gastown/CLAUDE.md BEFORE doing anything
  2. Write goal document with user stories
  3. Mail to Mayor: gt mail send mayor --subject "BUILD: <name>" --body "<goal>"
  4. Mayor runs pipeline (PRD → design → review → beads → sling)
  5. Relay human questions from Mayor to Theo. Mail answers back.
  6. When Mayor reports completion: deploy from merged main branch, verify.

Communication Bridge

Theo → Tycho (main session) → Mayor → Polecats ← Polecats ← Mayor (escalations/questions/completion) ← Tycho (relays to Theo)

Deploy Targets

  • CF Access: Applied to new projects for authentication
  • Pull from merged main branch, never from polecat worktrees

Build Lessons Learned

Build History & Lessons (click to expand)

Build 1: Outreach V2 (2026-03-29)

  • 3 beads (data + frontend + state), 2 parallel + 1 serial
  • First build that worked on first deploy
  • Lesson: x-cloak on body = white screen. Put on x-data div only.
  • Lesson: The plan is the product. Stress-tested plan = working app.
  • Lesson: Assembly was manual (Refinery didn't merge). Push data into rig repo BEFORE slinging.

Build 2: Tycho Dashboard (2026-03-29)

  • 2 beads (collector + frontend), both parallel
  • Collector returned nulls because spec was vague
  • Lesson: A spec is NOT a schema. A spec tells the agent exact paths, exact commands, exact error cases. If an agent has to guess, your spec failed.
  • Lesson: Don't skip from goals to beads. Goals → User Stories → Specifications → Contracts → Beads.
  • Lesson: Manually drove beads instead of using Mayor. Did the Mayor's job badly.

Build 2.1: Dashboard Iteration (2026-03-29)

  • Single bead rebuild with comprehensive spec
  • Warm polecat reused from build 2
  • Lesson: Detailed spec = working output. Vague spec = nulls.

Workspace

Rigs

gt rig add <name> <repo>   # Add project
gt rig list                # List projects
gt rig boot <name>         # Boot rig services

Mayor & Mail

gt mayor attach            # Start Mayor session
gt mail send mayor --subject "BUILD: X" --body "..."
gt mail inbox              # Check for escalations
gt nudge mayor "..."       # Quick message

Work Tracking

bd ready                   # Tasks with no open blockers
bd show <id>               # View task details
gt convoy create "Name" <ids...>  # Create convoy
gt convoy list             # List convoys
gt sling <bead> <rig>     # Assign work (Mayor does this)

Monitoring

gt feed                    # TUI dashboard

gt dashboard               # Web UI (port 8080)
gt peek <polecat>          # Watch agent work
gt agents                  # List active agents
gt polecat list <rig>      # List rig polecats

Memory & Learning

gt remember "lesson"                    # Store general memory
gt remember --type feedback "lesson"    # Store behavioral correction
gt memories                             # List all memories
gt memories --type feedback             # List corrections
gt seance                               # List dead sessions
gt seance --talk <id> -p "?"           # Query predecessor

Emergency

gt estop                   # Emergency stop all work
gt thaw                    # Resume after estop
gt scheduler pause/resume  # Pause/resume scheduler

Formulas

bd cook <formula>          # Execute formula
bd mol pour <formula>      # Create trackable instance
